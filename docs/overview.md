# 项目功能与处理流程说明

## 1. 这个项目是做什么的

这个项目是一个 **OpenHarmony / HarmonyOS ArkTS 应用隐私分析与可视化工具**。它会把一个 ArkTS 应用源码目录、一个 OpenHarmony SDK 源码目录，以及若干补充 CSV 数据作为输入，自动完成以下工作：

1. 扫描应用中的 ArkTS 页面、组件、生命周期函数和 SDK API 调用。
2. 找出可能涉及隐私数据处理的 **source**（数据进入点 / 生命周期入口）和 **sink**（调用系统能力或 SDK API 的位置）。
3. 构建 **调用图（Call Graph）**。
4. 从调用图中提取 **source → sink** 的路径，并进一步生成 **数据流（Dataflow）**。
5. 构建 **UI 树（UI Tree）**，识别页面、组件、交互节点和路由跳转。
6. 将数据流按 **页面 / 功能点** 重新聚合。
7. 为每个功能点抽取隐私事实，最终生成 **隐私声明报告**。
8. 提供 Web 可视化界面，方便从表格、图、功能点和最终报告多个视角查看结果。

换句话说，这个项目并不是单纯做“代码扫描”，而是在尝试回答三个更接近产品和合规的问题：

- 这个 App 在哪些地方读取、处理或展示了可能与隐私相关的数据？
- 这些数据是如何沿着代码流转到系统能力、页面交互或权限调用中的？
- 最终可以怎样整理成一份面向隐私声明撰写的结果？

## 2. 输入是什么

首页要求填写或选择 3 类核心输入，以及若干可选控制项：

### 2.1 必填输入目录

1. `appPath`
   - 待分析的 ArkTS 应用源码目录。
   - 默认值是 `input/app/Wechat_HarmonyOS/`。

2. `sdkPath`
   - OpenHarmony SDK 源码目录。
   - 默认值是 `input/sdk/default/openharmony/ets/`。

3. `csvDir`
   - SDK API 补充信息所在目录。
   - 默认值是 `input/csv/`。

### 2.2 可选控制项

1. `maxDataflowPaths`
   - 控制最多提取多少条 source → sink 路径。
   - 留空表示不限制。

2. 三组 LLM 配置
   - **数据流分析 LLM**：用于把调用路径扩展为带解释的数据流节点。
   - **UI 描述 LLM**：用于给 UI 节点生成人类可读名称。
   - **隐私报告 LLM**：用于从功能点数据流中抽取隐私事实，并组织成报告。

### 2.3 一个贯穿全文的例子

下面全文都用仓库里已经实际跑过的一次结果作为例子：

- App：`input/app/Wechat_HarmonyOS/`
- SDK：`input/sdk/default/openharmony/ets/`
- CSV：`input/csv/`
- 生成出的 `runId`：`Wechat_HarmonyOS_20260306-182228`

这次运行在 `output/Wechat_HarmonyOS/20260306-182228/` 下生成了结果，`meta.json` 中记录的统计信息是：

- 扫描 App 文件数：31
- sink 数：46
- source 数：29
- 调用图节点数：53
- 调用图边数：44
- 数据流数：26
- UI 树节点数：72
- 页面数：10
- 功能点数：12

## 3. 输入会经过哪些处理流程

后端主流程集中在 `server/src/app/run.ts` 中，可以按下面的顺序理解。

### 3.1 校验输入路径

系统会先把 `appPath`、`sdkPath`、`csvDir` 统一转成绝对路径，并检查它们是否存在、是否可读。

如果目录无效，分析会在最开始直接报错，不会产生后续结果。

### 3.2 创建输出目录和 runId

系统会根据应用目录名推断应用名，并拼出：

- `runId = <appName>_<时间戳>`
- `outputDir = output/<appName>/<时间戳>/`

例如本次例子中：

- 应用名：`Wechat_HarmonyOS`
- `runId`：`Wechat_HarmonyOS_20260306-182228`
- 输出目录：`output/Wechat_HarmonyOS/20260306-182228/`

### 3.3 构建 SDK 索引

系统会递归扫描 `sdkPath` 下的 `.d.ts` / `.d.ets` 文件，重点挑出文件名以 `@ohos.` 或 `@kit.` 开头的 SDK 声明文件，建立“模块名 → 文件路径”的索引。

这一步的意义是：后面在 App 代码里看到 `import xxx from '@ohos.xxx'` 或类似调用时，工具能知道这些符号对应哪个 SDK 模块。

### 3.4 扫描 App ArkTS 文件

系统只扫描应用目录下的：

- `entry/src/main/ets/`

并且只读取：

- `.ets`
- `.ts`

这意味着项目的分析重点是 ArkTS 主业务代码，而不是构建目录、产物目录或第三方依赖目录。

### 3.5 加载 CSV 补充信息

系统会读取 `csvDir` 中的补充 CSV，主要包括：

- SDK API 功能描述
- API 与权限之间的映射
- 覆盖 / 修正描述

这些 CSV 在整个分析链路里很关键，因为它们能把“某一行 SDK 调用”进一步解释成“这个 API 做什么”以及“可能关联哪些权限”。

### 3.6 分析 sinks：找出系统能力 / SDK API 调用

这一步会遍历 App 文件，做两件事：

1. 解析 SDK import 绑定。
2. 在源码里查找这些绑定对应的调用点。

对每一个命中的 sink，结果里会记录：

- App 源码文件路径
- 导入行号与导入代码
- 调用行号与调用代码
- API 功能描述
- 内部用的 API key / module 信息

可以把它理解成：

> “工具先把 App 中所有可能触达系统能力的位置找出来，并补上这些能力的中文语义说明。”

例如在 `Wechat_HarmonyOS` 的“我的”页面头像相关逻辑里，就会命中与媒体读取 / 写入相关的系统能力调用。

### 3.7 分析 sources：找出数据进入点 / 生命周期入口

source 的识别方式相对更轻量：工具会按行扫描函数定义，匹配一组预定义函数名，例如：

- `build`
- `aboutToAppear`
- `aboutToDisappear`
- `onPageShow`
- `onPageHide`
- `onBackPress`
- `onCreate`
- `onDestroy`
- `onForeground`
- `onBackground`
- `onWindowStageCreate`
- `onWindowStageDestroy`

这些函数在本项目里被视为“潜在数据进入点”或“业务逻辑入口点”。

例如：

- 页面 `build()` 往往是 UI 渲染和交互绑定的起点。
- `aboutToAppear()` 往往是页面初始化读取数据的起点。
- `onCreate()` / `onForeground()` 等则是应用生命周期相关逻辑的入口。

### 3.8 构建调用图（Call Graph）

调用图的核心思想是：

- 把扫描到的函数块变成节点；
- 把函数之间的调用关系变成边；
- 再把 sink 调用点挂接到其所在函数上；
- 把识别出的 source 标注到对应函数节点上。

最后工具会做一次裁剪，只保留那些 **真正位于 source → sink 路径上** 的节点和边。这样最终的图不会被无关函数污染太多。

对用户来说，这一步的价值是：

- 你能看到“哪一个生命周期 / 页面函数最终走到了哪个系统 API”；
- 也能看出中间经过了哪些业务函数。

### 3.9 提取 source → sink 路径

调用图建好后，系统会从所有 source 出发，寻找能到达 sink 的路径，并生成 path 列表。

这里 `maxDataflowPaths` 会生效：

- 如果你设置了 20，就最多保留 20 条路径；
- 如果留空，就尽量保留所有路径。

这一步相当于把“大图”切成若干条更容易理解的链路。

### 3.10 生成数据流（Dataflow）

这一步会把刚才提取出的路径交给 LLM 做更细的解释。每一条路径会被扩展成：

- 一串数据流节点
- 节点之间的边
- 每个节点的人类可读说明
- 一个 summary，概括数据项、处理目的、权限等信息

这里还会把源码上下文一并带进去，所以每个节点通常都包含：

- `filePath`
- `line`
- `code`
- `description`
- `context`

#### 退化行为

如果数据流 LLM 的 API Key 为空，这一步会被跳过，但系统仍然会继续生成：

- sink/source
- 调用图
- UI 树

只是 `dataflows.json` 会带有“skipped”标记，提示跳过原因。

### 3.11 生成 UI 树（UI Tree）

这一步的目标不是画出完整像素级界面，而是建立一个“页面—组件—交互—跳转”的结构化视图。

系统会：

1. 识别 `@Entry` 页面和 `/pages/` 下的主结构体。
2. 在 `build()` 里扫描组件起始行，例如按钮、输入框、图片等。
3. 识别交互事件，如 `.onClick(...)`。
4. 识别 `router` 跳转，并把“触发组件”与“目标页面”连起来。
5. 再结合启发式规则或 LLM，为节点补充更自然的标题。

因此 UI 树不只是“静态组件树”，还包含了页面之间的可导航关系。

### 3.12 按页面 / 功能点聚合数据流

有了 UI 树之后，系统会尝试把数据流重新组织成更符合产品视角的结构：

- 页面（Page）
- 页面下的功能点（Feature）
- 功能点下的数据流

这里会综合使用：

- UI 节点位置
- 页面结构体范围
- `build()` 函数范围
- source 命中位置
- 数据流节点与 UI 节点的行号距离

来判断“一条数据流更可能属于哪个页面、哪个功能点”。

例如在 `Wechat_HarmonyOS` 的输出里，`Mine` 页面被识别出一个功能点：

- `featureId = ui_Mine_4f14ddb6f2`
- `title = 头像`

该功能点下面聚合了 4 条数据流。

### 3.13 生成功能点隐私事实和最终隐私声明报告

当页面 / 功能点聚合完成后，系统会继续做两层产出：

#### 第一层：每个功能点的隐私事实

系统会读取该功能点的数据流，并生成 `privacy_facts.json`，里面通常会包含：

- `dataPractices`：收集了什么数据、来源是什么、如何处理、目的是什么
- `permissionPractices`：用了哪些权限、目的是什么、拒绝授权会怎样

这里有一个很重要的细节：

> 即使 LLM 抽取结果不完整，权限相关事实还会再结合 CSV 中的 API → 权限映射进行补强。

也就是说，权限部分不是纯 LLM 猜出来的，而是有一层确定性补充。

#### 第二层：最终隐私声明报告

系统会把所有功能点的隐私事实汇总，生成：

- `privacy_report.json`
- `privacy_report.txt`

最终报告分成两大部分：

1. 我们如何收集和使用您的个人信息
2. 设备权限调用

前端“隐私声明报告”页面展示的就是 `privacy_report.json`，其中某些文字还能直接点击并跳转到对应的数据流节点。

### 3.14 写入 run 注册表

分析结束后，系统会把本次 `runId` 写入：

- `output/_runs/<runId>.json`
- `output/_runs/latest.json`

这样前端如果不指定 `runId`，就可以默认读取最新一次分析结果。

## 4. 最终会输出哪些文件

仍以 `output/Wechat_HarmonyOS/20260306-182228/` 为例。

### 4.1 根目录下的核心文件

#### `meta.json`

记录本次运行的输入参数和汇总统计，例如：

- 输入目录
- 使用的 LLM provider / model
- 扫描文件数
- sink/source 数量
- 调用图、数据流、UI 树、页面、功能点数量

这是理解“这次分析总体规模”的最好入口。

#### `sinks.json` / `sinks.csv`

列出识别到的所有 sink 调用点。

适合回答：

- App 调用了哪些系统能力？
- 这些调用分别位于哪个文件、哪一行？
- 每个 API 的语义描述是什么？

#### `sources.json` / `sources.csv`

列出识别到的 source 入口点。

适合回答：

- 哪些生命周期函数、页面入口函数被当作数据流起点？

#### `callgraph.json`

保存裁剪后的调用图。

适合回答：

- source 是如何一步步流向 sink 的？
- 中间经过了哪些函数？

#### `dataflows.json`

保存全局的数据流结果，是“逐条路径解释版”的调用链。

适合回答：

- 这一条路径里，每个代码点扮演什么角色？
- 数据项是什么？处理目的是什么？

#### `ui_tree.json`

保存全局 UI 树，包括：

- 页面节点
- 按钮 / 输入框 / 组件节点
- 页面包含关系
- 跳转关系

#### `privacy_report.json` / `privacy_report.txt`

分别对应：

- 结构化报告数据
- 纯文本报告

如果你是要继续做前端展示、跳转或自动加工，优先看 `privacy_report.json`；如果是要直接阅读文案，优先看 `privacy_report.txt`。

### 4.2 `pages/` 目录下的分层结果

`pages/` 是这个项目很有价值的一层输出，因为它把“全局扫描结果”转成了“按产品页面组织的结果”。

#### `pages/index.json`

列出所有页面及其统计信息，例如：

- `AppLifecycle`
- `ComponentLogic`
- `Contact`
- `Home`
- `Mine`
- `SearchPage`

在例子里，`pages/index.json` 记录到：

- 页面数：10
- 功能点数：12
- 数据流数：26

#### `pages/<pageId>/ui_tree.json`

保存该页面裁剪后的 UI 树。

#### `pages/<pageId>/features/index.json`

列出页面下识别出的功能点。

例如 `Mine` 页下识别出了：

- `featureId = ui_Mine_4f14ddb6f2`
- `title = 头像`

#### `pages/<pageId>/features/<featureId>/dataflows.json`

保存某个功能点的数据流集合。

#### `pages/<pageId>/features/<featureId>/privacy_facts.json`

保存某个功能点抽取出来的隐私事实。

这份文件通常是“最终隐私声明报告”的直接上游。

### 4.3 运行索引文件

#### `output/_runs/latest.json`

保存最近一次运行的 `runId` 和输出目录。

#### `output/_runs/<runId>.json`

保存某一次运行的索引记录。

## 5. 会提供哪些可视化界面

前端是一个 React + Vite 页面，包含 1 个首页和 5 个结果页，共 6 个核心界面。

### 5.1 首页：分析配置与结果入口

首页是整个系统的控制台，主要功能有：

- 选择 / 输入 `appPath`、`sdkPath`、`csvDir`
- 配置 `maxDataflowPaths`
- 配置三组 LLM 参数
- 查看并选择已有 `runId`
- 发起分析任务
- 通过 SSE 查看进度阶段和百分比
- 跳转到各类结果页

首页上的可视化结果入口包括：

- 隐私声明报告
- sink API 信息
- source API 信息
- 调用图
- 数据流

### 5.2 sink API 可视化

这是一个表格页，展示 `sinks.json` 里的内容，重点字段包括：

- 文件路径
- 导入行号 / 调用行号
- 导入代码 / 调用代码
- API 功能描述

并且文件路径支持跳转到本地编辑器对应行。

### 5.3 source API 可视化

这也是一个表格页，展示 `sources.json` 中的 source 入口点，方便快速确认：

- 哪些生命周期函数被识别出来了
- 每个入口点位于哪里

### 5.4 调用图可视化

这个页面会把 `callgraph.json` 渲染成图，节点类型主要有：

- `source`
- `function`
- `sinkCall`

点击节点后可以看到：

- 文件位置
- 代码摘要
- 描述信息

如果本地编辑器路径可解析，还可以直接跳转到源文件。

### 5.5 数据流可视化

这是最重要的分析页之一。它提供了多层筛选：

- 选择页面
- 选择页面功能
- 选择具体数据流路径

选中某条路径后，页面会同时展示：

- 数据流图
- 当前选中节点的源码位置
- 当前节点的上下文代码片段
- 节点的人类可读说明

这让用户可以从“页面 / 功能”视角逐步下钻到具体代码行。

### 5.6 隐私声明报告页面

这个页面读取 `privacy_report.json`，把最终报告分成两段展示：

1. 我们如何收集和使用您的个人信息
2. 设备权限调用

它还有一个很实用的能力：

- 报告中的某些 token 带有跳转信息；
- 点击后会直接跳转到对应的“数据流可视化”页面与节点。

因此，这个页面不仅是“文案展示”，也是从合规文本回溯到代码证据的入口。

## 6. 用一个完整例子串起来：`Wechat_HarmonyOS` 的“头像”功能

下面用 `Mine` 页面里的“头像”功能把整条链路串起来。

### 6.1 从输入开始

本次运行输入如下：

- `appPath = input/app/Wechat_HarmonyOS/`
- `sdkPath = input/sdk/default/openharmony/ets/`
- `csvDir = input/csv/`

分析完成后生成：

- `runId = Wechat_HarmonyOS_20260306-182228`

### 6.2 页面与功能点识别

系统在 `pages/Mine/features/index.json` 中识别出 `Mine` 页面有一个功能点：

- 功能点标题：`头像`
- 功能点 ID：`ui_Mine_4f14ddb6f2`

这说明工具已经把“我的页面中的头像区域”识别成一个独立的可分析功能点，而不是散落在全局结果里。

### 6.3 数据流聚合

在 `pages/Mine/features/ui_Mine_4f14ddb6f2/dataflows.json` 中，可以看到该功能点下聚合了 4 条数据流。

其中一条数据流里可以看到这样的节点链条：

1. `@State imgPath: string = ""`
2. `build()`
3. `Column()`
4. `Image(this.imgPath ? this.imgPath : $r('app.media.ic_user_head1'))`
5. 点击事件 / 页面交互逻辑
6. 跳转或媒体选择相关逻辑

从这个结果可以看出，工具并不是只记录“有个头像图片”，而是进一步解释：

- `imgPath` 是头像路径状态；
- `Image(...)` 是头像数据的展示位置；
- 交互事件会把用户操作引到后续逻辑。

### 6.4 隐私事实抽取

在 `pages/Mine/features/ui_Mine_4f14ddb6f2/privacy_facts.json` 中，系统进一步整理出：

- 业务场景：用户在“我的”页面点击头像以选择并更新个人头像图片
- 数据来源：系统相册、应用本地状态变量
- 数据项：用户选择的图片 URI、媒体文件内容、头像图片路径 `imgPath`、应用包名等
- 处理方式：复制图片到应用缓存目录，并把路径存入组件状态用于 UI 渲染
- 处理目的：展示及更新用户头像信息

同时，权限部分还补出了：

- `ohos.permission.READ_MEDIA`
- `ohos.permission.WRITE_MEDIA`

并分别说明了：

- 权限用途是什么
- 拒绝授权后会造成什么影响

### 6.5 进入最终隐私声明报告

这些功能点级别的事实最后被汇总进 `privacy_report.txt` / `privacy_report.json`。

在本次例子里，最终报告中已经出现类似这样的内容：

- 用户在“我的”页面点击头像以选择并更新个人头像图片；
- 会收集用户选择的图片 URI、媒体文件内容、头像图片路径等；
- 会调用 `ohos.permission.READ_MEDIA` 与 `ohos.permission.WRITE_MEDIA`；
- 若拒绝授权，可能无法完成头像更新。

也就是说，这条链路已经从：

- 原始代码
- source / sink
- 调用图
- 数据流
- 页面功能点
- 功能点隐私事实
- 最终隐私报告

完整串起来了。

## 7. 可以怎样理解这个项目的输出价值

从不同角色看，这套输出有不同用法：

### 7.1 对研发

- 能快速定位隐私相关代码在哪个文件、哪一行。
- 能看到页面交互最终如何走到系统能力调用。
- 能回查某段报告文案到底来自哪一段代码证据。

### 7.2 对测试 / 安全 / 合规

- 能从页面和功能点视角查看隐私行为，而不是只看零散 API。
- 能把权限调用与业务场景对应起来。
- 能较快形成隐私声明初稿或审查依据。

### 7.3 对产品

- 能把“代码行为”翻译成“页面功能如何处理用户数据”的语言。
- 更容易判断某个权限申请是否真的与用户可见功能一致。

## 8. 一句话总结

如果把整个项目压缩成一句话，可以理解为：

> 它把 ArkTS App 源码、OpenHarmony SDK 和补充知识表结合起来，自动生成从代码证据到页面功能、再到隐私声明文案的一整套中间产物与可视化结果。

其中最关键的桥梁有三层：

- **调用图**：回答“代码怎么走”
- **页面 / 功能点聚合**：回答“这是哪个界面、哪个功能”
- **隐私报告**：回答“最终应该怎么对外说明”

因此，这个项目的核心价值不只是“检测 API”，而是把 **代码分析、界面理解、数据流解释和隐私合规表达** 串成一个可落地流程。
